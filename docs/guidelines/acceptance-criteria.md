# Acceptance Criteria Guidelines

Rules for writing acceptance criteria for user stories and tasks.

## General Rules

Every task must meet the following before being considered done:

- **Well documented:**
  - Code changes include clear comments where necessary.
  - Database modeling tasks include a Mermaid diagram in `docs/diagrams/`.
  - API endpoints are documented with request/response examples.

- **Well-described PR:** the pull request description must clearly explain what was done and why.

- **Peer reviewed:** all PRs must be reviewed and approved by at least one other team member before merging.

## Specific Acceptance Criteria

The specific acceptance criteria for each user story are written by the PO directly on the corresponding issue in the GitHub Projects board.
